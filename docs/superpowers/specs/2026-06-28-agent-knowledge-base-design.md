# Agent Knowledge Base — Design Spec

**Date:** 2026-06-28  
**Status:** Approved  
**Repo:** `wiki`

## Overview

Transform the `wiki` repository into an AI Agent-oriented knowledge base for curated summaries of excellent computer program design knowledge encountered in daily learning. The system uses Markdown files as the source of truth, a pre-built index for fast layered retrieval, and an MCP server as the formal read interface. Content ingestion is handled by Cursor Skills, not MCP write APIs.

## Design Decisions

| Dimension | Decision |
|-----------|----------|
| Consumption | Hybrid — Markdown files + MCP retrieval |
| Content form | Theme articles (~800–1500 words) |
| Organization | Hierarchical directories, AI-created organically (no fixed taxonomy) |
| Authoring | AI-assisted ingestion primary; occasional manual drafts reorganized later |
| Search | Layered retrieval (path/metadata filter → keyword); semantic search optional later |
| Metadata | Agent-optimized frontmatter |
| Input sources | Mixed — URLs, notes, conversation context |

## Architecture

### Principles

- **Git is the source of truth** — `knowledge/` Markdown is authoritative; `index/` is derived and rebuildable
- **Read/write separation** — MCP is read-only; writes go through Cursor Skills + Git
- **Agent-first** — frontmatter, article structure, and MCP response formats optimized for Agent consumption

### Repository Structure

```
wiki/
├── knowledge/                  # Knowledge articles (AI-created subdirectories)
│   └── {ai-created-path}/
│       └── {slug}.md
├── index/
│   ├── manifest.json           # Metadata index (generated by build-index)
│   └── embeddings/             # Reserved for future semantic search
├── schema/
│   └── entry.schema.json       # frontmatter JSON Schema
├── mcp-server/                 # TypeScript MCP Server (stdio)
│   ├── src/
│   │   ├── index.ts
│   │   ├── tools/
│   │   └── index-reader.ts
│   └── package.json
├── scripts/
│   └── build-index.ts          # knowledge/ → manifest.json
├── .cursor/skills/
│   ├── ingest/                 # Ingestion Skill (URL/notes/conversation → article)
│   └── reorganize/             # Reorganization Skill (draft → published article)
├── AGENTS.md                   # Agent usage guide (repo conventions)
└── README.md                   # Human-readable project description
```

### Data Flow

**Write path:** Input (URL / notes / conversation) → ingest or reorganize Skill → `knowledge/*.md` → build-index → `index/manifest.json`

**Read path:** Agent → MCP (`search_entries` / `list_tree`) → manifest → Agent → MCP (`get_entry`) → full article; Agent may also read files directly from workspace

### Conventions

| Convention | Rule |
|------------|------|
| File naming | `{slug}.md`, kebab-case, English or pinyin |
| Directory naming | kebab-case, AI-created based on content semantics |
| Article status | `draft` (pending reorganization) → `published` (archived) |
| Index update | ingest/reorganize Skills run `build-index` on completion |

## Frontmatter Schema

Every theme article requires YAML frontmatter, generated or completed by ingest/reorganize Skills:

```yaml
---
title: "Rust 错误处理：Result 与 ? 运算符"
date: 2026-06-28
summary: "Rust 用 Result<T,E> 显式处理可恢复错误，? 运算符简化传播链。"
status: published             # draft | published
tags:
  - rust
  - error-handling
keywords:
  - Result
  - "? operator"
  - error propagation
applies_to:
  - "可恢复错误的显式处理"
  - "库/API 的错误返回设计"
source:
  - url: "https://doc.rust-lang.org/book/ch09-00-error-handling.html"
    type: article             # article | video | book | conversation
related:
  - "languages/rust/ownership-and-borrowing.md"
---
```

### Field Constraints (`schema/entry.schema.json`)

| Field | Type | Required | Rules |
|-------|------|----------|-------|
| `title` | string | yes | 1–120 characters |
| `date` | string | yes | `YYYY-MM-DD` |
| `summary` | string | yes | 1–300 characters; Agent relevance check |
| `status` | enum | yes | `draft` \| `published` |
| `tags` | string[] | yes | at least 1 |
| `keywords` | string[] | yes | at least 1 |
| `applies_to` | string[] | yes | at least 1 |
| `source` | object[] | no | each item has `url` + `type` |
| `related` | string[] | no | paths must reference existing articles |

`build-index` validates frontmatter. MCP `search_entries` excludes `status: draft` by default (`include_drafts: true` to include).

### Article Body Template

Skills generate articles with fixed sections for Agent parsing:

```markdown
## 背景

## 核心思想

## 实践要点

## 代码示例

## 权衡与反模式

## 参考
```

- Target length: 800–1500 Chinese characters (or equivalent English)
- Code examples optional; "实践要点" and "权衡与反模式" required

### Manual Draft Minimum

```yaml
---
title: "临时标题"
date: 2026-06-28
summary: "待整理"
status: draft
tags: [uncategorized]
keywords: []
applies_to: []
---
```

Use reorganize Skill to complete fields, adjust directory, rewrite body, set `published`.

## MCP Server

### Tech Stack

| Item | Choice |
|------|--------|
| Language | TypeScript |
| Transport | stdio |
| SDK | `@modelcontextprotocol/sdk` |
| Index source | `index/manifest.json` loaded at startup |

Configured in Cursor via `mcp.json` pointing to local `mcp-server`.

### manifest.json Format

```json
{
  "version": 1,
  "generated_at": "2026-06-28T10:00:00Z",
  "entries": [
    {
      "path": "languages/rust/error-handling.md",
      "title": "Rust 错误处理：Result 与 ? 运算符",
      "date": "2026-06-28",
      "summary": "...",
      "status": "published",
      "tags": ["rust", "error-handling"],
      "keywords": ["Result", "? operator"],
      "applies_to": ["可恢复错误的显式处理"],
      "related": ["languages/rust/ownership-and-borrowing.md"]
    }
  ],
  "tree": {
    "languages": {
      "rust": ["error-handling.md", "ownership-and-borrowing.md"]
    }
  }
}
```

### Tools

#### `search_entries`

Layered retrieval core tool.

- **Parameters:** `query` (required), `path_prefix?`, `tags?`, `applies_to?`, `include_drafts?` (default false), `limit?` (default 10, max 50)
- **Returns:** matched entries with `path`, `title`, `summary`, `score`, `matched_fields`
- **Flow:** filter by path/tags/applies_to → keyword scoring on title/summary/keywords/tags/applies_to → sort by score desc

#### `get_entry`

- **Parameters:** `path` (relative to `knowledge/`)
- **Returns:** `{ frontmatter, body, related_entries: [{path, title, summary}] }`

#### `list_tree`

- **Parameters:** `path?` (default root)
- **Returns:** `{ directories: [...], entries: [{path, title, summary}] }`

#### `get_stats`

- **Parameters:** none
- **Returns:** `{ total_entries, published, drafts, top_level_dirs, last_indexed }`

### Semantic Search Extension (Future)

When enabled, write vectors to `index/embeddings/`. Add `mode: "semantic" | "keyword"` parameter to `search_entries` (default keyword). MCP tool signatures unchanged.

### Error Handling

| Scenario | Behavior |
|----------|----------|
| manifest.json missing | Clear error; suggest `npm run build-index` |
| manifest stale (source mtime newer) | Warn at MCP startup; use stale index |
| get_entry path not found | 404-style error + similar path suggestions (Levenshtein) |
| frontmatter validation failure | build-index errors and exits; no index written |

## Skills and Index Building

### Skill Responsibilities

| Skill | Trigger | Input | Output |
|-------|---------|-------|--------|
| ingest | Active ingestion | URL / notes / conversation | New `.md` + updated manifest |
| reorganize | Draft cleanup | `status: draft` files | Completed frontmatter, moved path, `published` status |

### ingest Flow

1. Identify input type (URL / notes / conversation)
2. Extract or fetch source content
3. Generate theme article draft
4. Determine directory path and slug (kebab-case)
5. Dedup check against manifest (similar title/tags → prompt user: update / new / cancel)
6. Write `knowledge/{path}/{slug}.md`
7. Run `build-index`
8. Output summary: path, title, tags

### reorganize Flow

1. Read target file (user-specified or scan all drafts)
2. Complete/correct all frontmatter fields
3. Rewrite body to match template (fill missing sections)
4. Move file if directory inappropriate; update `related` references
5. Set `status: published`
6. Run `build-index`

### build-index

```bash
npm run build-index
```

Steps:

1. Recursively scan `knowledge/**/*.md`
2. Parse YAML frontmatter
3. Validate against JSON Schema (fail with file/field details)
4. Validate `related` paths exist
5. Generate `entries` array and `tree` structure
6. Write `index/manifest.json`

Triggers:

- Automatically after ingest/reorganize Skill completion
- Optionally via Git pre-commit hook (deferred)
- MCP startup: warn if stale, do not auto-rebuild

| Step | Deliverable |

### Deferred (YAGNI)

- Semantic search / embeddings
- Web UI
- Git pre-commit hook
- Multi-user / permissions
- CI/CD

## Evolution Roadmap

| Phase | Trigger | New Capabilities |
|-------|---------|------------------|
| Phase 1 | Now | Files + MCP + Skills (this spec) |
| Phase 2 | >50 articles or keyword search insufficient | Vector embeddings, semantic search |
| Phase 3 | Maintenance cost rises | pre-commit auto-rebuild, dead link check, duplicate alerts |
| Phase 4 | Use outside Cursor | Remote MCP server or read-only mirror |

## Testing Strategy

| Layer | Coverage |
|-------|----------|
| build-index unit tests | frontmatter parsing, Schema validation, tree generation |
| MCP integration tests | 4 tool input/output contracts |
| End-to-end smoke | example article → build-index → MCP search → get_entry |

Framework: Vitest.

## Non-Goals

- Fixed taxonomy management UI
- Web rendering/browsing of articles
- MCP write/update/delete tools
- i18n for metadata field names (content language free; field names English)
